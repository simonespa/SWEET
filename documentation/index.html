Description [index] [next->]

MicroWSMO Editor is a semantic descriptor of RESTfull web services. With this editor the users can semantically annotates all the web services, simply using a graphical approach. This editor works with the hREST microformat

Approach's Overview [<--prev] [index] [next->]
  • Initially only the service hREST tag is enabled, because it is the root element of the hREST schema. If we adopt this strategy, we can avoid which other kind of hREST tag will be used for first. Subsequently the use of service, also label, method, address and operation hREST tags will be enabled. After the use of operation, we can enable input and output. This way of proceed, is in step with the hREST microformat constraint.
  • Every time the user tags a part of the document with an hREST element, an XML file will be generated. This file describe the hREST annotation inside the web page (how the hREST elements are indented each other). A parser will parse this XML file behind the scene, and will check if this document is valid according to an XMLSchema compliant to the hREST microformat. It will work at run-time, after every tag event.
  • Another panel will visualize the hREST schema document by a tree view, in order to stand out the hREST element hierarchy. Basically, it provide the graphical visualization of the XML file mentioned above. This allow the user to check the structure just created, to edit or delete the element of the schema, by a context menù.
  • IMORTANT: the validator will checks if a specific portion of code have already tagged with the same hREST tag. If true, in this case will be shown a message box that will warn the user about this mistake.
    NOTE: every time the user tags a portion of text, a random ID is automatically generated and is associated to its.
  • The web service description can be tagged by the hREST tree structure (by a double click on the item of the tree), or by the context menù, which will appear when the user right click on the highlighted text. This redundancy is very effective, and make the editor more flexible and friendly for every kind of users.
  • There will be a menù bar, where the user will be able to do the operation showed above and more:
    1. Open new web service descriptions into the viewer, inserting the URL
    2. Change the look of the editor
    3. Consult an howto, online help, or screencast about "how annotate a RESTfull web service".
G.U.I. [<-prev] [index] [next->]
  • The starter page looks like the most popular IDE, such as Eclipse or NetBeans for example.
  • On the left side, there is the control panel. This panel is divided in two tab. A tab is for the hREST annotation, while the other is for the semantic annotation by the ontologies.
  • On the right side, there is an html viewer, which is wider than the left side panel. This viewer show the web service description, and allow the user to annotate it, by the tools of the editor.
  • At the bottom of the editor, under the html viewer, there is a collapsible panel that will show all the informations and details each time the user click onto an item of the editor, in order to help him and advise about the procedures to follow.
  • Over the editor there is a component which contain a menu bar while under this menu bar (over the html viewer) there is a text field. Into this text field the user can inserts the URL of a web service description.

The Highlights [<-prev] [index] [next->]
  1. When the user tags a portion of the web page, the system have to garantee the HTML code integrity. To be more precise, if the user highlights a part of text that is already tagged by a couple of html tag, the system have to avoid the overlap in order to keep the document well formed. For example:

    Annotation example (HTML code)

    The service's address is <code>http://happenr.3scale.ws/webservices </code>

    if the user selects a part of text over another couple of tag, crossing them

    Annotation example (HTML code)

    The service's address is<code>http://happenr.3scale.ws/webservices </code>

    without any expedient, we could obtain the following result

    Annotation example (HTML code)

    The service's <span class="some_hREST_tag">address is <code>http://happenr.3scale.ws </span>/webservices</code>

    The service's <span class="some_hREST_tag">address is <code>http://happenr.3scale.ws </span>/webservices</code>

    In order to avoid the problem described above, the system can warns the user by a message box.
Future Features [<-prev;] [index]
  • A tab next the html viewer. This panel will show the html code of the web page, so the expert user will be able to insert manually the hREST tags.
  • The Ontologies Visualization Page. It will be a view where the user can see the details of a given ontology. All the elements of this ontology will be showed in a tree view. On the left side will be a panel which will contains a tab for the visualization of the ongology, and probably other tabs to show different views of the given ontology. On the right side will be a panel which two tabs. One tab will show the details of the selected item into the ontology, while the other will show a graph which will represent the ontology.

    NOTE: this features will be placed probably in another html page, and will be invoked by a GET method from the main editor page.
